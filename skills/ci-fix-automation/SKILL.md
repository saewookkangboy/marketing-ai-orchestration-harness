# CI Fix Automation

## Purpose

실패한 CI 체크를 빠르게 진단하고, 재현 가능한 수정 루프로 안정화한다.

## Triggers

- "CI 깨졌는데 고쳐줘"
- "실패 로그 보고 원인 찾아줘"
- "체크 통과할 때까지 반복해줘"
- "GitHub Actions 실패 수정해줘"

## Inputs

- 필수: 워크플로우 실행 정보(브랜치/PR/런 ID)
- 필수: 실패 체크 범위(전체/특정 잡)
- 선택: 수정 허용 범위(테스트만/코드 포함)
- 선택: 시간 제한, 반복 횟수 제한

## Workflow

1. 실패 맥락 수집
   - 실패한 잡, 스텝, 에러 메시지, 최초 실패 커밋을 정리한다.
2. 원인 가설 수립
   - 환경 이슈/테스트 취약성/코드 회귀/의존성 문제로 분류한다.
3. 로컬 재현
   - 가능한 범위에서 동일 테스트/빌드 커맨드를 재실행한다.
4. 최소 수정 적용
   - 원인과 직접 연결된 최소 변경을 적용하고 부작용을 점검한다.
5. 재검증
   - 관련 테스트 및 타입/린트 확인 후 결과를 요약한다.
6. 루프 종료 판정
   - 해결 시 종료, 미해결 시 다음 가설로 1회전 더 수행한다.

## Safety Rules

- 로그에 포함된 토큰/비밀값을 출력/커밋하지 않는다.
- CI 외부 요인(외부 서비스 장애)은 코드 변경으로 우회하지 않는다.
- 스냅샷/골든 업데이트는 승인 없이 대량 변경하지 않는다.

## Output Contract

- 실패 원인 요약(확정/추정 구분)
- 적용 수정과 근거
- 실행한 검증 커맨드와 결과
- 남은 리스크 및 다음 액션

## Example Invocation

"현재 PR의 GitHub Actions 실패 체크를 분석하고, 로컬 재현 가능한 범위에서 최소 수정으로 통과 전략을 적용해줘."
